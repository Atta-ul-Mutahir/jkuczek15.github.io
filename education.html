    Ethics

    In many courses I took at Iowa State, we were given example situations where we were forced to make a difficult ethical decision. One case study about a scandal with a Therac-25 radiation therapy machine caught my interest. Patients were given radiation therapy treatments that were potentially hundreds of times greater than the dose assigned for them. If you're interested, you can find the case study here. As Software Engineers, it is our responsibility to ensure the safety of the users using the software. In the next few paragraphs, I'll discuss how I would've handled the issue.

    An obvious thing to do was to test the machine. It is difficult to replicate the scenario where there is a 1-byte counter and it overflows at the exact time the user manually enters input into the machine. With a machine that carries that much impact on medical patients, it is important that every critical part of the machine is tested. This means they should run the program through hundreds of tests with expect dosages for different patients. If any of the tests fail, the product is not ready to be moved into a production phase. The root cause of the problem was that the software was designed in such a way that it was “realistically impossible to test in a clean and automated way”. If the product was tested properly, the concurrent programming bugs could have been fixed prior to the software being moved into a production phase.

    I think laziness is a big part of why the engineers did what they did. They probably had to deal with some sort of legacy system that wasn’t worth upgrading to perform automated tests. It could also be the case that the engineers had false confidence that their work was perfect, though, it wasn’t. You shouldn’t ever think your work is perfect in Software Engineering, when in doubt you should always test. I think from the start they didn’t have a good way to test the system. Instead of an engineer coming in to develop a new way of testing (boring and takes a lot of time), most engineers just want to be innovative and build the future product. Professionals want to write great code, amateurs want to impress you with features.
